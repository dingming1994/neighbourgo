## Codebase Patterns
- ChatModel uses Freezed for code generation; after changing model fields, run `dart run build_runner build --delete-conflicting-outputs`
- Chat IDs are deterministic: task chats use `{taskId}_{sorted_uids}`, direct chats use `direct_{sorted_uids}`
- ChatRepository is provided via Riverpod `chatRepositoryProvider`; screens access it with `ref.read(chatRepositoryProvider)`
- Navigation uses GoRouter with `context.push(AppRoutes.chatThread.replaceFirst(':chatId', chatId))`
- PublicProfileScreen gets userId from GoRouter path params, current user from `currentUserProvider`
- Pre-existing warnings/infos in `flutter analyze` are acceptable; only errors block commits
- BidListSection requires both `taskId` and `posterId` params; posterId is needed for chat creation between poster and bidder
- Use `FutureProvider.family` for lightweight one-off data fetches (e.g., provider rating) rather than streaming full profiles

## Codebase Patterns
- Integration tests use `IntegrationTestWidgetsFlutterBinding.ensureInitialized()` and live in `integration_test/`
- Use `initializeTestApp()` from `test_helpers.dart` to connect to Firebase emulators (Auth:9099, Firestore:8080, Storage:9199)
- Use `signInTestUser(email:, password:)` to create/login test users, `seedUser()` to write Firestore user docs
- Bids are stored as subcollections: `tasks/{taskId}/bids/{bidId}` — use `firestore.collection('tasks').doc(taskId).collection('bids')` to seed/read
- Chat IDs are deterministic: `{taskId}_{sortedUids.join('_')}`
- After bid seeding, also update `bidCount` on the task doc so UI reflects it
- Use `.last` when tapping dropdown items (e.g., `find.text('Ang Mo Kio').last`) to hit overlay, not trigger
- Welcome screen has 'Continue with Email' button → EmailAuthScreen with email/password fields → role selection → profile setup
- `settle(tester, seconds: N)` pattern: pump N seconds then pumpAndSettle with 15s timeout
- Existing unit tests have 2 pre-existing failures in auth_screens_test.dart — not caused by integration test changes
- SubmitBidSheet shows two 'Submit Bid' texts (title + button) — use `.last` to tap the button
- Task detail role logic: `isPoster = currentUser.uid == task.posterId`, `isProvider = !isPoster` — not based on user role field
- After programmatically updating Firestore (seeding acceptance/completion), navigate away and back to force UI refresh
- Cross-user Firestore writes in tests: use `seedBidViaRest()` / `seedDocViaRest()` with `Authorization: Bearer owner` header
- GoRouter route order matters: specific routes (`/tasks/post`) must come before parameterized routes (`/tasks/:taskId`)
- Chat field name in Firestore is `participantIds` (matches security rules), not `participants`
- Emulator data clear: use project ID `neighbourgo-sg`, e.g. `curl -X DELETE "http://localhost:8080/emulator/v1/projects/neighbourgo-sg/databases/(default)/documents"`

## Codebase Patterns
- Integration tests live in `integration_test/` at project root, using Flutter's `integration_test` SDK package
- Firebase emulator ports: Auth=9099, Firestore=8080, Storage=9199, Functions=5001 (defined in `firebase.json` and mirrored in `integration_test/firebase_emulator_config.dart`)
- Models use Freezed with `fromJson` factories; Firestore deserialization via `XModelExt.fromFirestore(doc)` static extensions
- State management: Riverpod (`flutter_riverpod`), routing: GoRouter with ShellRoute
- Firestore collection names are in `lib/core/constants/app_constants.dart` (e.g. `usersCol`, `tasksCol`)
- User roles: `poster`, `provider`, `both` — stored as string in Firestore
- Task statuses: `open`, `assigned`, `inProgress`, `completed`, `cancelled`, `disputed`
- All 242 Sprint 4 unit/widget tests must remain green
- Dev login navigates to `/auth/role` (role selection) not `/home` — modified in welcome_screen.dart
- Router redirect exempts `/auth/role` and `/auth/profile-setup` so logged-in users can complete onboarding
- Integration tests pump `ProviderScope(child: NeighbourGoApp())` directly — no need to call `main()`
- For splash screen timing: use `pump(Duration(seconds: 3))` then `pumpAndSettle()` to wait for 2s delay + navigation
- Use `find.widgetWithText(TextFormField, 'hint text')` to target specific form fields by their hint text
- For dropdown selection: tap the `DropdownButtonFormField<String>`, wait for overlay, then `find.text('Option').last` to tap the overlay item
- Scroll before interacting with off-screen widgets: `tester.drag(find.byType(SingleChildScrollView), Offset(0, -300))`
- Bids are a Firestore subcollection: `tasks/{taskId}/bids/{bidId}` — use `firestore.collection('tasks').doc(taskId).collection('bids')` to query
- Multi-user integration tests: create both users in setUpAll, use `signOutTestUser()` / `signInTestUser()` + re-pump the app widget to switch users
- Chat IDs are deterministic: `{taskId}_{sortedUid1}_{sortedUid2}` — construct manually or use `ChatRepository.buildChatId()`
- Chat seeding requires: chat doc in `chats/`, task (status=assigned), accepted bid in `tasks/{id}/bids/`, both user docs
- Accepting a bid auto-creates a chat document — no need to manually seed chats after `acceptBid()` in end-to-end tests
- Task completion: "Mark as Complete" button visible when poster + status is `assigned` or `inProgress`; calls `completeTask()` which sets status='completed' + completedAt
